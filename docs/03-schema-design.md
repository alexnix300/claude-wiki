# Step 3 — The Schema (CLAUDE.md)

## Context

The schema is the most important file in the system. It lives at the root of the wiki vault as `CLAUDE.md` (for Claude Code) or `AGENTS.md` (for Codex). It tells the LLM how to behave as a wiki maintainer — what conventions to follow, what formats to use, what workflows to execute.

Without the schema, the LLM is a generic chatbot. With it, the LLM is a disciplined wiki maintainer that follows consistent conventions across sessions.

## What the schema contains

### 1. Directory structure

A map of every directory and its purpose. The LLM references this to know where to file things.

### 2. Page types

Each wiki directory has a defined page type with a clear purpose:

| Type | Directory | Purpose |
|------|-----------|---------|
| Source summary | `wiki/sources/` | One page per ingested document |
| Entity | `wiki/entities/` | People, orgs, clients, systems |
| Concept | `wiki/concepts/` | Ideas, frameworks, patterns |
| Topic | `wiki/topics/` | Thematic synthesis across sources |
| Analysis | `wiki/analyses/` | Filed query results, investigations |
| Project | `wiki/projects/` | Personal and client projects |
| Personal | `wiki/personal/` | Routines, goals, habits, life |
| Research | `wiki/research/` | Tech exploration, learning notes |

### 3. Page format

Every page uses YAML frontmatter + Obsidian markdown:

```markdown
---
type: source | entity | concept | topic | analysis | project | personal | research
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [list of source filenames]
tags: [relevant, tags]
---

# Page Title

Content with [[wikilinks]] for cross-references.

## See also

- [[Related Page]]
```

### 4. Cross-referencing rules

- Use `[[wikilinks]]` for all internal links
- Every page must have at least one inbound link
- When updating a page, check if related pages need new links

### 5. Index and log conventions

- `wiki/index.md` — catalog organized by page type, one-line entry per page
- `wiki/log.md` — append-only, each entry: `## [YYYY-MM-DD] operation | Subject`

### 6. Workflows

Three core workflows defined: Ingest, Query, Lint. See [[06-workflows]] for details.

### 7. Style guidelines

- Concise, no filler
- Bullet points over prose
- Specific claims over vague summaries
- Explicit contradiction handling
- Source attribution via wikilinks

## The actual file

The complete schema file is at the root of the vault: `CLAUDE.md`. It was generated by the agent based on the use case (general-purpose work/projects/knowledge base) and refined through conversation.

## Comment — Schema evolution

The schema is not static. It evolves as you use the wiki:

- **First week**: You'll discover page types you need that aren't defined. Add them.
- **First month**: You'll find conventions that are too strict or too loose. Adjust.
- **Ongoing**: As the wiki grows, you may need new workflows (e.g. "archive", "merge pages", "split page").

The right approach: when something feels wrong, tell the LLM to update the schema. The schema should always reflect how you actually want the wiki to work, not how you imagined it would work on day one.

## Prompt used

No separate prompt — the agent generated the schema automatically after the initial setup questions (Step 1). The schema was then expanded when additional categories (projects, personal, research) were requested:

> "can we add to global claude.md to use wiki across the sessions... creating structure for: personal routines and chats, projects - personal / clients, research, etc."
